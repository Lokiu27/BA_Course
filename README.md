# Репозиторий контента курса по бизнес-анализу

Этот репозиторий содержит **только Markdown-файлы** документации курса. Вся инфраструктура (конфигурация MkDocs, Dockerfile, настройки nginx) находится в основном репозитории [bolotova.online](https://github.com/Lokiu27/bolotova.github.io).

Готовый сайт доступен по адресу: **https://bolotova.online/course/**

---

## Структура файлов

```
course-content/
├── README.md        ← этот файл (инструкции для авторов)
└── index.md         ← стартовая страница курса
```

### Правила именования файлов

- Используйте только строчные латинские буквы, цифры и дефисы: `tema-1.md`, `modul-2-analiz.md`
- Не используйте пробелы, кириллицу или специальные символы в именах файлов
- Главная страница раздела — всегда `index.md`
- Для группировки тем создавайте подпапки: `modul-1/tema-1.md`

## Как добавлять и редактировать материалы

### Добавление новой страницы

1. Создайте файл с расширением `.md` в корне репозитория или в подпапке
2. Напишите содержимое в формате [Markdown](https://www.markdownguide.org/basic-syntax/)
3. Сделайте коммит и отправьте изменения в ветку `main`

Пример структуры страницы:

```markdown
# Заголовок темы

Вводный текст раздела.

## Подраздел

Содержимое подраздела.

> **Примечание:** используйте блоки цитат для важных замечаний.
```

### Редактирование существующей страницы

1. Откройте нужный `.md` файл
2. Внесите изменения
3. Сделайте коммит и отправьте в ветку `main`

### Что можно добавлять

- Markdown-файлы (`.md`) с текстом, заголовками, списками, таблицами, блоками кода
- Изображения (`.png`, `.jpg`, `.svg`) — размещайте в папке `assets/images/`
- Ссылки на внешние ресурсы

### Что добавлять **нельзя**

Этот репозиторий — только для контента. Не добавляйте:

- `mkdocs.yml` — конфигурация MkDocs находится в основном репозитории
- `Dockerfile`, `docker-compose.yml` — инфраструктура в основном репозитории
- Конфигурации nginx, CI/CD-скрипты или любые другие инфраструктурные файлы
- Файлы `.github/workflows/` (кроме `trigger-build.yml`, который уже настроен)

## Процесс автоматической сборки

После отправки изменений в ветку `main` сайт обновляется автоматически:

```
Вы делаете push в main
        ↓
GitHub Actions запускает workflow trigger-build.yml
        ↓
Отправляется repository_dispatch в основной репозиторий
        ↓
Основной репозиторий запускает пересборку Docker-образа
        ↓
Docker клонирует этот репозиторий и запускает mkdocs build
        ↓
Обновлённый сайт публикуется на https://bolotova.online/course/
```

Обычно от коммита до публикации проходит 3–5 минут. Статус сборки можно отслеживать во вкладке **Actions** основного репозитория.

### Как это работает технически

В этом репозитории настроен workflow `.github/workflows/trigger-build.yml`, который при каждом push в `main` отправляет событие `repository_dispatch` в основной репозиторий через секрет `DISPATCH_PAT`. Основной репозиторий получает событие, запускает `docker compose up --build`, клонирует этот репозиторий и собирает MkDocs-сайт.

Если сборка завершается с ошибкой (например, из-за некорректного Markdown), текущая версия сайта остаётся без изменений, а в Actions появится уведомление об ошибке.
